# Comparar Funciones de Plataformas (Product Research)

Esta habilidad permite realizar una investigación de producto comparando cómo diferentes plataformas resuelven una funcionalidad específica. Su objetivo es extraer aprendizajes de UX, modelos de datos y flujos de trabajo para informar el diseño de las features de Kanarii.

## Instrucciones

### 1. Selección de Referencias
Para cada análisis, selecciona entre **3 y 5 plataformas**, buscando una mezcla equilibrada:
- **La Referencia Grande**: Un estándar de la industria (ej. Notion, Slack, GitHub).
- **La Referencia Simple**: Una app enfocada en la simplicidad (ej. Todoist, Bear).
- **La Referencia Nicho**: Una plataforma cercana al caso de uso específico o con una propuesta innovadora.

### 2. Dimensiones de Análisis
El análisis debe ir más allá de la interfaz visual (UI). Para cada plataforma, investiga:
- **Modelo Mental**: ¿Cómo entiende el usuario la función? ¿Es un documento, una tarea, un nodo, un evento?
- **Secuencia de Acciones**: ¿Cuál es el "happy path" desde el inicio hasta el resultado?
- **Datos Mínimos**: ¿Qué información es estrictamente necesaria para que la función sea útil?
- **Automatización**: ¿Qué partes del flujo son automáticas y cuáles quedan a cargo del usuario?

### 3. Restricción de Gobernanza (Kanarii)
**CRÍTICO**: Kanarii no sigue modelos de jerarquía corporativa tradicional.
- **ALERTA**: Si una plataforma analizada utiliza un modelo de control centralizado en un "superadmin invisible" o jerarquías rígidas, señálalo explícitamente como **incompatible**.
- **FOCO**: Busca patrones que promuevan la autonomía, la transparencia y la gobernanza distribuida.

### 4. Output de la Skill
El reporte debe estructurarse en estos tres bloques:

#### 📊 Plataformas Revisadas
Breve resumen de las apps seleccionadas y por qué se eligieron.

#### 🧠 Patrones Detectados
Síntesis de los modelos mentales, flujos y arquitecturas de información encontrados.

#### 🎯 Traducción a Kanarii
Recomendaciones accionables divididas en:
- **MVP**: Qué implementar de inmediato.
- **Backlog**: Qué dejar para más adelante.
- **Validación**: Qué dudas de UX debemos validar con los usuarios reales.

## Restricciones
- No te limites a describir capturas de pantalla; analiza la arquitectura de la funcionalidad.
- Si una solución es "estándar" pero tóxica para el modelo de Kanarii, propón una alternativa disruptiva.
